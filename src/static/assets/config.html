Layout Type
Controls how the cells are arranged in the collection view. After selecting the layout type here, you may further adjust the layout settings by selecting the appropriate layout configuration page on the left hand side.
Table Layout
A basic layout implementation, the Table Layout will lay out its elements as a vertically scrolling list where each row can have either a fixed height or a variable height.

Optionally, the table layout may also display headers and footers for each section in the collection view.

Flow Layout
The flow layout arranges cells in a grid-like vertically scrolling container. The cells may have a fixed or variable size.

Optionally, the flow layout may also display headers and footers for each section in the collection view.
Masonry Layout
The masonry layout arranges cells in a vertically scrolling container. Cells will be laid out in equally wide columns and each new cell will be positioned in the column with the topmost space available, starting from the leftmost cell.

Additionally, columns in the masonry layout may have a speed assigned to them which affects the scrolling for that column alone.

The masonry layout does not support grouping by sections or displaying headers and footers.
Stack Layout
The stack layout is a vertically scrolling layout that presents cells as a stack, where the current cell appears above the other cells. In the stack layout, previous cells appear behind the current cell, while upcoming cells are hidden.

When scrolling in the stack layout, the scroll position will always snap back to fully show a single cell.

The stack layout does not support sections or supplementary views.
Tile Layout
The tile layout is a vertically scrolling layout that arranges cells as they appear in the data set, wherever there is space for them within the visible area, preferring positions that are closer to the top-left corner of the area.

To help with achieving a well aligned layout, tile layout allows setting a reference unit size and spacing between cells. Tile layout will automatically assign multiples of the unit size to cells, also taking the spacing size into account.

Optionally, the tile layout may also display headers and footers for each section in the collection view.

ADDITIONAL INFORMATION
Handling events
Upon choosing a data source, collection view creates one binding source for each field in that data set. The purpose of these binding sources is to make it easy to interact with events.

Whenever any event is triggered for a cell, all of the binding sources from the data set are updated to match the value of the cell that triggered the event. Note that it is only guaranteed that these values are correct only while the event is being handled.
For example, if you bind the double-click event to an action that shows the details of the double-clicked item and then click on a different item, even if the click doesn't trigger any action, the values of the event bindings will be modified to match the newly clicked item.
Advanced
NOTE: These properties primarily deal with lower-level performance optimizations and using them incorrectly may lead to unexpected behavior. You should not change these properties if the collection view does not have any performance issues.
TESTED ADVANCED FEATURES
Off-screen Buffer
When scrolling quickly, the renderer may not be able to keep up with adding and removing items from the collection view, which may lead to items missing from the edges of the visible area or the collection view contents flashing.

To help combat this, the collection will render elements off-screen so that they are already rendered when its container scrolls.
Increasing this value will cause more items to be rendered off-screen, increasing the initial rendering time but decreasing the likelihood of unexpected behavior while scrolling.
Decreasing this value will increase the initial rendering time, but is likely to cause temporarily flashing items while scrolling.

When using iScroll this value is ignored and always assumed to be 0.

iScroll is automatically enabled when using a scroll bar style, the stack or masonry layout or when pinned headers or footers are enabled.

Use iScroll on Windows Desktop
Causes iScroll, a custom scrolling implementation to be used instead of native scrolling on Windows desktops. This will cause the collection to take control of the entire scrolling process, eliminating all issues related to flashing items while scrolling, but it may have a negative impact on scrolling performance in certain configurations.
Additionally, the usual system scrollbars will be replaced with a custom iOS-like scrollbar.
Always Use iScroll on iOS
Causes iScroll, a custom scrolling implementation to be used instead of native scrolling on all iOS browsers. This will cause the collection to take control of the entire scrolling process, eliminating all issues related to flashing items while scrolling, but it may have a negative impact on scrolling performance in certain configurations.

This should be used when the cell mashups have more complex contents. In certain cases, this can cause a jittering behavior while scrolling as the renderer is unable to keep up with the increased load; Enabling this property will resolve that issue on iOS devices.

Note that while running in an UIWebView, the collection view will always use iScroll to be able to react to all scroll events.
Resize Responsive Widgets
This property should be enabled if the cell mashups contain responsive widgets that need to perform additional work during resizes (for example charts and gauges) and the cells can be resized if the size of the collection view changes (for example when using flow layout with the flow setting set to expand).

Normally, to improve performance, collection view does not invoke the resize method on the cell mashup. This works fine for most mashups, but in certain cases it may lead to visual artifacts after the cell is resized. In those cases, this property should be enabled to resolve those issues at a minor cost in performance.

Note that when the root widget of the cell mashups is a View widget, this property has no effect as resizing is handled by CoreUI instead of the usual Thingworx methods.
EXPERIMENTAL ADVANCED FEATURES
Fast Widget Append
When this property is enabled, the usual widget rendering method is replaced for collection view cells with a faster version that skips certain steps in the widget initialization process. This will cause certain complex mashups (especially those with nested layouts) to render faster in some configurations, reducing the initial rendering time.

Widgets created through this method will never have bounding boxes or outer containers thereby reducing the number of elements added to the document, however this method has not been fully tested with all widgets and may produce unexpected side-effects when the cell mashups contain certain widgets.
